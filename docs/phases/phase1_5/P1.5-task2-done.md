## What Was Implemented

### New AIS Friends service
Created `backend/app/services/ais_friends_service.py` with:
- `AisFriendsService` class
- async `fetch_ships(bbox: dict | None = None)` method
- module-level `fetch_ships()` wrapper for repo consistency

### Request behavior
The service:
- calls `https://www.aisfriends.com/api/public/v1/vessels/bbox`
- uses `httpx.AsyncClient`
- sends `Authorization: Bearer <token>`
- uses a 30 second timeout
- defaults to a full-world bounding box when `bbox` is omitted:
  - `lat_min=-90`
  - `lat_max=90`
  - `lon_min=-180`
  - `lon_max=180`
- retries once on HTTP 429 before returning an empty list
- logs and returns `[]` when the API key is missing

### Normalization behavior
The service normalizes AIS Friends vessel payloads into the repo's existing `Ship` contract:
- `id`
- `lat`
- `lon`
- `heading`
- `speed`
- `name`
- `destination`
- `ship_type`
- `timestamp`

Primary spec field handling:
- `mmsi` -> `id`
- `lat` -> `lat`
- `lon` -> `lon`
- `true_heading` -> `heading`
- `speed_over_ground` -> `speed`
- `name` -> `name`
- `destination` -> `destination`
- `ship_type` -> `ship_type`
- `timestamp` -> normalized ISO timestamp

Defensive fallback handling also supports likely AIS Friends doc variants:
- `latitude` / `longitude`
- `course_over_ground` for heading fallback
- `reported_name`
- `ais_destination`
- `type` / `detailed_type`

### Ship type mapping
The service maps upstream ship type values into the repo's existing stable categories:
- `cargo`
- `tanker`
- `passenger`
- `fishing`
- `other`

String and numeric ship type inputs are both handled conservatively.

### Error handling
The service logs and returns `[]` on:
- missing API key
- request timeout
- HTTP errors
- invalid JSON / parse failure
- unexpected top-level payload shape
- missing or invalid `data` list
- repeated rate limiting after one retry
